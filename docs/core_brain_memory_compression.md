# 核心大脑：代理循环与记忆压缩机制深度解析

在一个长达数小时的终端 Debug 会话中，AI 会执行几十次命令行、阅读上百个文件。如果只是简单地将所有的历史信息塞给模型，任何大语言模型的上下文窗口都会在十分钟内被撑爆，甚至导致模型“失忆”或“胡言乱语”。

为了解决这个问题，系统底层的代理循环（Agent Loop）并不是一个简单的对话框，而是一个极其复杂的“**记忆管理器**”。它设计了四重防线，通过丢弃、折叠、总结，硬生生在有限的上下文里挤出了无限的续航能力。

---

## 代理主循环：事件驱动的思考机器

整个系统的运作围绕着一个被称为“查询循环”的核心函数展开。
你可以把它想象成一个永不疲倦的工厂流水线：

```javascript
// 简化的代理循环伪代码
async function 代理主循环(人类输入) {
  // 把人类的话加入历史记忆
  历史记忆.推入(人类输入);
  
  while (true) {
    // 每次发给模型前，先进行记忆清理
    清理后的记忆 = 执行四层记忆压缩(历史记忆);
    
    // 发送给模型，等待它的决定
    模型决策 = 等待大模型回复(清理后的记忆);
    
    if (模型决策.想要使用工具) {
      // 比如模型想用 "读取文件" 工具
      结果 = 本地执行工具(模型决策.参数);
      历史记忆.推入(结果); // 把结果塞进记忆，开启下一轮循环
      continue;
    } 
    
    if (模型决策.是自然语言对话) {
      // 任务完成，模型开口说话了
      打印到终端(模型决策.内容);
      break; // 跳出循环，等待人类下一次输入
    }
  }
}
```

在这套循环里，最核心、最具技术含量的部分，就是它在发送给模型前执行的**四层记忆压缩（Context Collapse）**。

---

## 第一重防线：结果截断与隐藏缓存（Tool Result Budget）

这是防范模型“自杀”的最前线。
假设大模型执行了一个 `npm install` 命令，或者读取了一个几万行的日志文件。如果把这几万行直接塞进记忆，模型瞬间就会爆掉。

**处理策略**：
1. **掐头去尾**：系统对每一次工具的返回结果设置了预算（比如最多 4000 个字符）。如果超出，它会强行切断，只保留结果的最前面一小部分（通常是命令或文件头）和最后面一小部分（通常是报错堆栈的末尾）。中间那几万字直接被丢弃。
2. **兜底缓存**：那被丢弃的内容真的没用了吗？系统并没有彻底删除它们，而是把完整的几万字悄悄写入了本地硬盘的一个临时文件里（例如 `/tmp/claude-xxx.log`）。
3. **注入引导提示**：最后，在发给模型的消息里，系统会在被截断的地方强行打上一块“补丁”：

> *[系统提示：中间的巨量输出已被截断以节约空间。如果以上的截断内容不足以让你排查问题，请你自己调用文件读取工具，去阅读本地缓存文件 `/tmp/claude-xxx.log` 获取完整内容。]*

**效果**：模型不会被垃圾信息撑死，但如果它真的需要看中间的报错细节，它又被赋予了“去指定地点看完整监控”的自主权。

---

## 第二重防线：思考气泡剪枝（Snip Compact）

为了让模型在解决复杂问题时更有条理，系统鼓励模型在调用工具前先输出一段 `<thinking>我该怎么做呢？</thinking>` 的心理活动。
这在最近的两三轮对话中是非常有用的上下文。但在 50 轮对话之后，模型昨天是怎么思考的，对于今天的任务已经毫无帮助了。

**处理策略**：
剪枝算法会定期巡视历史记忆。当发现某些思考气泡距离当前上下文已经足够遥远时，它会像园丁剪除枯枝一样，把这些 `<thinking>` 标签和里面冗长的心理活动直接抹除。
对于模型而言，它只需要记得自己“曾经调用过某个工具，拿到了什么结果”，至于当时的心理博弈，不需要再占用宝贵的脑容量。

---

## 第三重防线：微观塌缩与打扫战场（Microcompact）

这是一个非常天才的清理机制，专门用来对付“无用的中间产物”。

大模型为了找一个 Bug，可能连续用了 10 次搜索命令，翻阅了 20 个文件的目录。当它终于找到目标并修改了代码后，前面那 10 次的搜索结果就全成了垃圾。

**处理策略**：
“微观塌缩”算法会去分析历史记忆的依赖关系。如果它发现有一大段长文本的工具返回结果，在随后的几轮对话中再也没有被模型提及或利用过，它就会判定这是一块“已经被废弃的战场”。
它会将这段几千字的冗长结果直接压缩成一行简短的墓碑（Tombstone）：

> *[系统提示：此处的历史执行结果已被折叠，以优化上下文性能]*

通过这种方式，系统能把前面累积的十几万无用记忆瞬间释放，让大模型的注意力死死盯在当前正在解决的 Bug 上，彻底根除了 AI 常见的“聊久了就变蠢”的问题。

---

## 第四重防线：启动子模型写回忆录（Autocompact）

如果前面三招都用尽了，面对那种聊了一整天、真的快要顶破模型物理极限的马拉松级对话，该怎么办？
系统拿出了最后的终极杀招：**宏观记忆总结**。

**处理策略**：
当系统检测到总记忆量即将触及危险红线时，它会在主循环之外，偷偷在后台唤醒另一个速度极快、价格便宜的小模型。
系统会把最古老的那几十轮对话喂给这个小模型，并给它下达一个任务：

> *“请阅读以下冗长的人机对话，用500字总结他们在这个项目里排查了什么 Bug，最终得出了什么结论，保留所有出现过的关键变量名、文件路径和系统环境状态。”*

拿到这 500 字的摘要后，主进程会痛下杀手，将那几十轮原始对话硬生生从记忆中删除，只把这段摘要塞进去。

**效果**：就像人脑一样，对于很久以前的事情，我们不再记得每一句原话，只保留核心的认知。这种机制赋予了系统理论上“无限长度”的会话续航能力。

---

**总结**：一个优秀的终端智能体，其核心竞争力根本不在于调用 API，而在于这套极度精密的“记忆垃圾回收机制”。这四重防线，保证了大模型永远处于最敏锐、最清醒的工作状态。
